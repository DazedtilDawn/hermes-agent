# Lane Recommendation Memory

This layer exists to stop the system from rediscovering the same optimization lessons every hour.

## Purpose

Burn intelligence and cause hypotheses can be correct in the moment but still remain tactical. Lane recommendation memory promotes repeated patterns into durable candidates for policy, routing, or process change.

## What it watches

- repeated lane burn insights across burn-attribution snapshots
- repeated burn-cause patterns across governance incidents

## What it produces

- durable lane policy candidates
- repeated burn-cause candidates for process correction
- recurring optimization recommendations that deserve explicit review rather than ad hoc repetition

## Why it matters

Without memory, the system can notice the same waste repeatedly without becoming wiser. With memory, recurring patterns become candidates for:
- permanent cheap-first routing
- protected-lane preservation
- retry/cron cleanup
- routing threshold changes
- policy updates

## Guardrail

Recommendation memory should never silently rewrite policy. It should promote recurring evidence into reviewable candidates, not convert repetition into automatic authority.
